# الدرس 02: كيف تبني Gap Analysis محترم

## لماذا هذا الدرس مهم

لأن Gap Analysis هو المكان الذي يتحول فيه فهمك للنظام وللعميل إلى حكم استشاري مهني. إذا كان الـ Gap Analysis ضعيفًا، فإما أن تضخم الفجوات بلا داعٍ، أو تخفي فجوات حقيقية حتى تنفجر بعد Go-Live.

## الهدف من الدرس

بعد هذا الدرس يجب أن تكون قادرًا على:

- تعريف معنى Gap في مشاريع ERP بشكل صحيح.
- التمييز بين ما يحل بالتدريب وما يحل بالإعداد وما يحتاج تطويرًا.
- كتابة Gap بصيغة تساعد القرار لا بصيغة إنشائية.
- ربط كل Gap بالأثر والأولوية وطريقة المعالجة.

## الشرح المفاهيمي الأساسي

Gap ليست أي فرق بين ما يريده المستخدم وما يعرفه عن النظام.

Gap حقيقية تعني:

هناك احتياج عملي أو رقابي أو تشغيلي مهم، والنظام في وضعه القياسي الحالي لا يغطيه كما يجب.

### ليست كل مشكلة Gap

| الحالة | هل هي Gap؟ |
| --- | --- |
| المستخدم لا يعرف الإجراء القياسي | غالبًا لا، بل Training issue |
| البيانات غير مكتملة | غالبًا لا، بل Data readiness issue |
| الإعداد الحالي لا يعكس الواقع ويمكن تعديله | غالبًا Configuration issue |
| يوجد احتياج لا يدعمه النظام القياسي فعلًا | نعم، هذه Gap محتملة |

### ما الذي يجب أن تحتويه Gap جيدة؟

- وصف واضح للحالة الحالية
- وصف واضح للاحتياج
- سبب عدم التغطية
- الأثر التشغيلي أو الرقابي أو المالي
- الأولوية
- التوصية: Configuration أو Process change أو Report أو Development

## كيف يظهر المفهوم في الأنظمة الثلاثة

### Dynamics 365

في Dynamics تظهر الفجوات كثيرًا عند محاولة فرض features متقدمة أو عند الحاجة إلى تقارير أو تدفقات خاصة لا يغطيها standard process كما هو. المستشار الجيد هنا يفرق بين ميزة غير مفعلة وميزة غير مدعومة أصلًا.

### Oracle Fusion Cloud

في Oracle يظهر الـ Gap Analysis بشكل قوي جدًا في التقارير، UAT، Process exceptions، وتعريف ما إذا كانت الحاجة تحل عبر Work Definition أو Parameter أو Costing setup أو تحتاج تدخلًا تطويريًا. Oracle يفرض انضباطًا أعلى في ربط gap بالتصميم.

### SAP S/4HANA

في SAP كثير من ما يبدو Gap في البداية يكون في الحقيقة ضعف فهم لـ standard process أو ضعف Master Data. لكن توجد أيضًا gaps حقيقية في التقارير أو التكامل أو السلوك المطلوب. وهنا تصبح صياغة الفجوة بدقة أمرًا حاسمًا.

## منظور المستشار Functional Consultant View

اسأل نفسك قبل تسجيل أي Gap:

- هل فهمت الاحتياج الحقيقي أم مجرد رغبة المستخدم الحالية؟
- هل جرّبت تفسير standard process بشكل كامل أولًا؟
- هل المشكلة في الإعداد أم في البيانات أم في التدريب؟
- إذا كانت Gap حقيقية، فما أثرها لو لم تُحل؟
- هل تحتاج Development أم يكفي تقرير أو Workflow أو تعديل إجراء؟

## الأخطاء الشائعة Common Mistakes

- تسجيل كل اختلاف على أنه Gap.
- كتابة Gap بصيغة عامة غير قابلة للقرار.
- تجاهل الأثر والأولوية.
- القفز سريعًا إلى التطوير Development قبل استنفاد البدائل.
- عدم ربط Gap بسيناريو UAT أو owner واضح.

## المراجعة السريعة Quick Review

- ليست كل مشكلة Gap.
- فرق كبير بين Training issue وConfiguration issue وReal gap.
- Gap الجيدة يجب أن تكون قابلة للقرار والتنفيذ.
- أفضل مستشار هو من يقلل الضباب لا من يزيده.

## مهمة الدرس Study Task

اكتب ثلاث أمثلة:

- مثال ليس Gap بل Training issue
- مثال ليس Gap بل Data readiness issue
- مثال Gap حقيقية تستحق تقريرًا أو تطويرًا
