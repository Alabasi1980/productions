# 02_Oracle_Work_Order_Cycle.md
# دورة أمر العمل Work Order Lifecycle في Oracle Fusion Cloud Manufacturing

## 1. مقدمة

دورة أمر العمل في Oracle Fusion Cloud Manufacturing لا تبدأ من شاشة **Work Order** فقط، بل تبدأ من طبقات مترابطة يجب أن تكون جاهزة قبل التنفيذ. في Oracle، أمر العمل هو نتيجة لتعريفات سابقة تشمل الأصناف، هياكل المنتج، مراكز العمل، الموارد، العمليات، وتعريف طريقة الإنتاج.

تعتمد الدورة على الطبقات التالية:

- **Master Data**: الأصناف، المنظمات، المستودعات، المواقع، الموارد، مراكز العمل، الورديات، ومعاملات المصنع.
- **Item Structure / BOM**: هيكل المواد الداخلة في المنتج.
- **Work Definition**: القالب الذي يجمع Item Structure مع Operations وResources وOutputs.
- **Operations**: خطوات التصنيع.
- **Resources**: العمالة، المعدات، أو الموارد المستخدمة في العمليات.
- **Work Centers**: وحدات الإنتاج التي تنفذ العمليات.
- **Work Order**: الأمر التنفيذي لإنتاج كمية معينة.
- **Material Issue**: صرف المواد إلى أمر العمل.
- **Operation Execution**: تنفيذ العمليات وتسجيل الإنجاز.
- **Product Completion**: إكمال واستلام المنتج النهائي.
- **Cost Accounting**: معالجة تكلفة الحركات.
- **Work Order Close**: الإغلاق التشغيلي والمالي.
- **Variances**: الانحرافات الناتجة عند الإغلاق، خصوصًا في Standard Cost.
- **Period Close**: إغلاق الفترة المحاسبية بعد التأكد من معالجة أوامر العمل.

تفصل Oracle بين أربع طبقات مهمة:

| الطبقة | المعنى | سبب أهميتها |
|---|---|---|
| **Work Definition** | تصميم طريقة الإنتاج | يحدد كيف يجب أن يُنتج المنتج قبل إنشاء الأمر. |
| **Work Order Execution** | تنفيذ أمر العمل | يسجل الصرف، العمليات، الموارد، والإكمال. |
| **Inventory Transactions** | حركات المخزون | تحول الأحداث التشغيلية إلى أثر مخزني قابل للتتبع. |
| **Cost Accounting** | معالجة التكلفة | يسعر الحركات ويولد Cost Distributions قبل الوصول للمحاسبة. |
| **Subledger / General Ledger** | الترحيل المحاسبي | يحول أحداث التكلفة إلى قيود محاسبية حسب قواعد SLA وGL. |

النتيجة المهمة: Oracle لا يتعامل مع الإنتاج كعملية واحدة، بل كسلسلة أحداث مترابطة؛ كل حدث قد يكون له أثر على التنفيذ، المخزون، WIP، التكلفة، والمحاسبة.

## 2. النظرة العامة على دورة أمر العمل

الدورة النصية المختصرة:

```text
1. إعداد Master Data
2. إعداد Item Structure / BOM
3. إعداد Work Areas / Work Centers / Resources
4. إعداد Work Definition
5. إنشاء Work Order
6. حالة Unreleased
7. Release Work Order
8. حالة Released
9. Issue Materials
10. Execute Operations
11. Resource Charging
12. Material Return إن وجد
13. Product Completion
14. حالة Completed
15. Work Order Close
16. حالة Closed
17. Cost Accounting / Variances
18. Period Close / GL Transfer
```

| المرحلة | الهدف | المخرجات الأساسية |
|---|---|---|
| إعداد Master Data | تجهيز الأصناف، المنظمات، الموارد، والمستودعات | بيانات صالحة لإنشاء Work Definition وWork Order |
| إعداد Item Structure / BOM | تحديد المواد الداخلة في المنتج | مكونات المنتج وكمياتها |
| إعداد Work Areas / Work Centers / Resources | تجهيز بيئة التنفيذ والموارد | مراكز عمل وموارد قابلة للاستخدام والجدولة |
| إعداد Work Definition | تحديد طريقة إنتاج المنتج | قالب إنتاج يجمع العمليات والمواد والموارد والمخرجات |
| إنشاء Work Order | تحويل التصميم إلى أمر تنفيذي | أمر عمل بحالة Unreleased مع عمليات ومواد وموارد منسوخة |
| Unreleased | مراجعة الأمر قبل التنفيذ | أمر قابل للتعديل وغير مسموح بالحركات عليه |
| Release Work Order | إطلاق الأمر للتنفيذ | أمر ظاهر للمشغلين وقابل للصرف والتنفيذ |
| Released | تنفيذ الأمر | إمكانية صرف مواد وتسجيل موارد وإكمال عمليات |
| Issue Materials | صرف المواد لأمر العمل | WIP Material Issue وحركة مخزون وتكلفة لاحقة |
| Execute Operations | تسجيل تقدم العمليات | إكمال عمليات حسب Count Point / Auto-Transact / Optional |
| Resource Charging | تسجيل استخدام العمالة أو الآلات | تكلفة موارد تضاف إلى WIP |
| Material Return | عكس صرف مواد زائدة أو خاطئة | زيادة المخزون وخفض WIP |
| Product Completion | استلام المنتج النهائي | زيادة Finished Goods وخفض WIP بتكلفة تقديرية |
| Completed | اكتمال العملية الأخيرة | أمر جاهز للإغلاق لكنه لم يُسوَّ ماليًا بالكامل |
| Work Order Close | حسم التكلفة والتسوية | Actual Cost / Variances / إغلاق WIP |
| Closed | إنهاء الدورة | منع التعديل وإتمام الحوكمة |
| Cost Accounting / Variances | معالجة وتوزيع التكلفة | Cost Distributions وانحرافات أو تعديلات تكلفة |
| Period Close / GL Transfer | التأكد من اكتمال الفترة | منع انتقال فروقات الإنتاج لفترة خاطئة |

## 3. مرحلة ما قبل أمر العمل — Master Data Readiness

قبل إنشاء Work Order، يجب أن تكون بيانات الإنتاج الأساسية مضبوطة. ضعف هذه البيانات يؤدي مباشرة إلى أوامر إنتاج خاطئة، حركات مخزون غير صحيحة، تكلفة غير دقيقة، أو تعطل في الإغلاق.

| البيان الأساسي | دوره في دورة أمر العمل | أثر الخطأ فيه |
|---|---|---|
| **Items** | تمثل المواد الخام، المنتجات نصف المصنعة، والمنتجات النهائية | صنف غير مضبوط يؤدي إلى صرف أو استلام أو تسعير خاطئ. |
| **Raw Materials** | مواد تُصرف إلى أمر العمل | خطأ خصائصها يؤثر على Material Issue وWIP. |
| **Semi-Finished Goods / Subassemblies** | قد تكون منتجات وسيطة لها Work Definitions مستقلة أو Phantom | عدم وضوحها يسبب تضخم أو نقص في أوامر العمل الفرعية أو الصرف. |
| **Finished Goods / Assembly** | المنتج النهائي المطلوب تصنيعه | خطأ تعريفه يمنع Work Definition أو Product Completion الصحيح. |
| **Item Structure** | يحدد مكونات المنتج | خطأ المكونات أو الكميات يؤدي لصرف خاطئ وانحرافات. |
| **UOM** | وحدة قياس الأصناف والموارد | وحدة غير دقيقة تسبب كميات أو أوقات أو تكاليف غير منطقية. |
| **Inventory Organization** | المنظمة التي يتم فيها الإنتاج والمخزون | عدم تعريفها كبيئة تصنيع يمنع أو يربك تشغيل Manufacturing. |
| **Manufacturing Plant** | منظمة مخزون مفعلة كـ Plant عبر Plant Parameters | دونها لا تعمل وظائف التصنيع بالشكل المطلوب. |
| **Plant Parameters** | تحدد سلوك المصنع مثل Process Manufacturing وPrevent Issue of Expired Lots | إعداد خاطئ يؤثر على الصرف، الإكمال، الهالك، وطريقة العمل. |
| **Subinventory** | مستودع منطقي للصرف أو الاستلام | خطأه يسبب صرف أو استلام في موقع غير صحيح. |
| **Locator** | موقع تفصيلي داخل Subinventory | خطأه يضع المخزون في رف أو خانة غير صحيحة. |
| **Work Area** | منطقة إنتاج داخل المصنع | عدم وجودها يمنع تنظيم Work Centers. |
| **Work Center** | وحدة تنفيذ العمليات | خطأه يؤثر على العمليات، الموارد، والجدولة. |
| **Workstation** | موقع مادي محدد داخل Work Center | يؤثر على تسجيل المشغلين وResource Instances عند الحاجة. |
| **Resource** | عمالة أو معدات تدخل في العملية | مورد غير مضبوط يسبب تكلفة أو جدولة غير صحيحة. |
| **Standard Operation** | عملية قياسية تُعاد استخدامها | ضعفها يؤدي إلى تكرار وتعريف غير موحد للعمليات. |
| **Shift / Calendar** | يحدد وقت العمل والطاقة المتاحة | خطأه يجعل الجدولة غير واقعية. |
| **Supply Type** | يحدد طريقة صرف المادة: Push/Pull/Bulk/Phantom/Supplier | خطأه قد يؤدي إلى صرف زائد، عدم صرف، أو Backflush غير مرغوب. |
| **Lot / Serial Control** | يحدد تتبع اللوت أو السيريال | خطأه يضعف التتبع والجودة وGenealogy. |
| **Costing Enabled** | يسمح بتسعير الصنف | إذا لم يكن مضبوطًا قد لا تظهر تكلفة صحيحة. |
| **Inventory Asset Value** | يحدد ما إذا كان المخزون ذا قيمة أصلية | خطأه يؤثر على تقييم المخزون والمحاسبة. |

## 4. مرحلة Item Structure / BOM

يمثل **Item Structure / BOM** في Oracle هيكل المنتج الذي يحدد المواد الداخلة في المنتج النهائي. لا يعمل وحده كقالب إنتاج كامل، بل يُستخدم داخل **Work Definition** ليصبح جزءًا من العملية التنفيذية.

### دوره في الدورة

- يحدد المواد الداخلة في المنتج.
- يُستهلك من قبل Work Definition لتكوين **Operation Items**.
- يحدد ما يجب صرفه عند تنفيذ Work Order.
- يؤثر على كمية Material Issue.
- يؤثر على Cost Accounting لأن تكلفة المواد هي جزء أساسي من WIP وتكلفة المنتج.
- عند تغيير Item Structure، يمكن مزامنة التغيير إلى Work Definitions المتأثرة، مع إخطار الجهات المعنية إذا كانت الصلاحية مفعلة.

| مفهوم Item Structure / BOM | أثره على دورة Work Order | أثره على التكلفة |
|---|---|---|
| **Item Structure** | يحدد مواد المنتج التي تنتقل إلى Work Definition | مصدر Material Cost الأساسي. |
| **BOM Components** | تتحول إلى Operation Items داخل Work Definition | أي نقص أو زيادة يؤدي إلى Material Usage أو Substitution Variance. |
| **Item Structure Synchronization** | يزامن التغييرات مع Work Definitions المتأثرة | يمنع استخدام تعريفات قديمة في Cost Rollup أو Work Orders. |
| **Subassemblies** | قد تنتج بأمر مستقل أو تظهر كـ Phantom | تؤثر على عدد الأوامر أو تفجير المكونات داخل الأمر. |
| **Ad hoc Items** | يمكن إضافتها إلى Work Definition بصلاحية خاصة | قد تسبب اختلافات إذا لم تضبط حوكمة إضافتها. |
| **Supply Type** | يحدد طريقة صرف المكون | يؤثر على توقيت Material Issue وWIP. |

### مخاطر ضعف Item Structure قبل إنشاء Work Order

| الخطر | الأثر |
|---|---|
| مكونات ناقصة | لا تُصرف مواد مطلوبة، وتظهر تكلفة ناقصة أو إنتاج غير قابل للتنفيذ. |
| مكونات زائدة | صرف مواد غير مطلوبة وتضخم WIP وتكلفة المنتج. |
| كميات غير صحيحة | انحرافات Material Usage. |
| عدم مزامنة التغييرات | Work Definition تستخدم بنية قديمة لا تمثل الواقع. |
| عدم وضوح المواد Phantom أو Supplier | أخطاء في الصرف أو Outside Processing. |

## 5. مرحلة Work Definition

**Work Definition** هو القالب المركزي في Oracle Manufacturing. هو ليس BOM فقط، بل يجمع بين هيكل المنتج، العمليات، الموارد، المخرجات، الإصدارات، الأولويات، والتواريخ.

في Oracle، Work Definition يعرف طريقة تصنيع المنتج، ويُستخدم كقالب لإنشاء Work Order.

### عناصر Work Definition

| عنصر Work Definition | دوره | أثره على Work Order | أثره على التكلفة |
|---|---|---|---|
| **Header** | يحتوي Work Method، الصنف، الأولويات، وموقع الإكمال | يحدد بيانات الأمر الأساسية عند النسخ | يؤثر على Cost Rollup وCompletion Location. |
| **Operations** | تسلسل خطوات الإنتاج | تُنسخ كعمليات في Work Order | تحدد مواضع الموارد والمواد. |
| **Operation Items** | المواد المطلوبة لكل عملية | تحدد ما يُصرف ومتى | مصدر Material Cost وMaterial Variances. |
| **Operation Resources** | العمالة أو المعدات المطلوبة | تحدد Resource Charging | مصدر Resource Cost وResource Variances. |
| **Operation Outputs** | مخرجات العملية في Process Manufacturing | تحدد المنتج الأساسي والمنتجات المشتركة/الثانوية | تؤثر على توزيع التكلفة. |
| **Primary Work Definition** | الطريقة الأساسية للإنتاج | يستخدمها التخطيط عادة | قد تستخدم في Lead Time وCost Rollup. |
| **Alternate Work Definition** | طريقة بديلة للإنتاج | تتيح مواد أو خطوط أو عمليات بديلة | قد تستخدم Costing Priority مختلفة. |
| **Work Definition Version** | إصدار بتاريخ سريان | يحدد أي تعريف يُستخدم وقت إنشاء الأمر | يحافظ على دقة التكلفة التاريخية. |
| **Effective Start / End Date** | صلاحية زمنية للتعريف | تمنع استخدام تعريف خارج فترة صلاحيته | تؤثر على Cost Rollup وأوامر تاريخية. |
| **Production Priority** | أولوية الإنتاج | تحدد Primary/Alternate من منظور الإنتاج | تؤثر على اختيار التخطيط. |
| **Costing Priority** | أولوية التكلفة | قد تسمح باستخدام تعريف مختلف للتكلفة | تؤثر مباشرة على Cost Rollup. |

### لماذا Work Definition أهم من BOM وحدها في Oracle؟

لأن BOM يجيب على سؤال: **ما المواد المطلوبة؟**

أما Work Definition فيجيب على أسئلة أوسع:

- ما المواد المطلوبة؟
- في أي عملية تُستهلك؟
- ما مركز العمل الذي ينفذ العملية؟
- ما الموارد المطلوبة؟
- ما المخرجات؟
- ما الإصدار الصحيح؟
- ما تعريف التكلفة المستخدم في Cost Rollup؟

لذلك، في Oracle، Work Definition هو أساس Work Order، وليس مجرد قائمة مواد.

## 6. إنشاء Work Order — Create Work Order

**Work Order** هو التمثيل التنفيذي لإنتاج كمية محددة من صنف معين في تاريخ معين. عند إنشائه، تتحول Work Definition من قالب إلى أمر قابل للتنفيذ.

| البند | التوضيح |
|---|---|
| ما هو Work Order؟ | أمر تنفيذي يحدد الصنف، الكمية، Work Definition، تواريخ البداية والنهاية، والعمليات المطلوبة. |
| طرق الإنشاء | يدويًا من Manage Work Orders، أو من Supply Chain Planning، أو Order Management / Orchestration، أو Min-Max Planning، أو REST API / FBDI. |
| البيانات المطلوبة | Item، Quantity، Start أو Completion Date، Work Definition، Inventory Organization / Plant. |
| ما يُنسخ من Work Definition | Operations، Operation Items، Operation Resources، وربما Outputs في Process. |
| الأثر المخزني | لا توجد حركة مخزون فعلية بعد، مع احتمال وجود Reservations في سيناريوهات معينة. |
| الأثر التكلفي | لا قيد تكلفة بعد، لكن الأمر يصبح Cost Object يستقبل الحركات لاحقًا. |
| المخاطر | اختيار Work Definition أو Version غير مناسب أو تاريخ سريان خاطئ. |

| الأثر | ماذا يحدث عند إنشاء Work Order؟ |
|---|---|
| التشغيل | إنشاء أمر قابل للمراجعة والتعديل قبل التنفيذ. |
| المخزون | لا حركة فعلية، وقد توجد حجوزات في حالات محددة. |
| WIP | لا يزيد WIP بعد. |
| التكلفة | لا Cost Distribution بعد، لكن الأمر يصبح وعاءً للحركات المستقبلية. |
| المحاسبة | لا أثر محاسبي مباشر. |
| المخاطر | Work Definition خاطئة تنتج أمرًا خاطئًا من بدايته. |

## 7. حالة Unreleased

حالة **Unreleased** هي الحالة المعتادة عند إنشاء Work Order. في هذه الحالة يكون الأمر موجودًا، قابلًا للتعديل، لكنه غير جاهز للتنفيذ.

| البند | التوضيح |
|---|---|
| المعنى | أمر عمل مُنشأ ولم يُطلق للتنفيذ. |
| ما يسمح به | تعديل تفاصيل الأمر ومراجعة العمليات والمواد. |
| ما يمنع | لا تُسمح حركات تنفيذ مثل صرف مواد أو تسجيل موارد أو إكمال عمليات. |
| الأثر المخزني | لا حركة مخزون فعلية. |
| الأثر التكلفي | لا تكلفة فعلية بعد. |
| الانتقال | يمكن الانتقال إلى Released أو Canceled. |

### مخاطر ترك Work Order في Unreleased بدون متابعة

- أوامر مخططة لا تتحول إلى تنفيذ.
- انقطاع بين التخطيط والإنتاج.
- طلبات قد تظهر كأنها مغطاة وهي فعليًا لم تبدأ.
- **استنتاج وظيفي مبني على منطق المقال:** يجب وجود تقرير أو متابعة لأوامر Unreleased المتأخرة.

## 8. Release Work Order — حالة Released

عند **Release** يتحول Work Order إلى أمر جاهز للتنفيذ. يظهر في **Dispatch List** للمشغلين، ويصبح قابلًا لصرف المواد، تنفيذ العمليات، تسجيل الموارد، أو التعامل مع Outside Processing عند الحاجة.

| الأثر | ماذا يحدث في Released؟ |
|---|---|
| التشغيل | يصبح الأمر متاحًا للمشغلين في Dispatch List. |
| المخزون | لا يعني أن المواد صُرفت، لكنه يسمح بحركات الصرف. |
| WIP | لا يزيد WIP بمجرد Release؛ يبدأ الأثر مع الصرف أو الموارد. |
| التكلفة | الأمر يصبح جاهزًا لاستقبال حركات تُسعّر لاحقًا. |
| المحاسبة | لا قيد مباشر بمجرد Release، لكن الحركات التالية ستنتقل إلى Cost Accounting. |
| المخاطر | Release قبل توفر المواد أو قبل ضبط البيانات يؤدي لتعطل التنفيذ أو صرف خاطئ. |

### Validations مستخرجة من المقال

| Validation | السبب |
|---|---|
| لا Release قبل جاهزية البيانات الأساسية | لأن Work Order يعتمد على Work Definition والمواد والموارد. |
| التأكد من وجود On-hand مناسب للمواد عند التنفيذ | Oracle يذكر ضرورة التأكد من توفر رصيد للمواد قبل تنفيذ الأمر. |
| التعامل مع Pending Approval إذا كانت E-Records/E-Signatures مفعلة | لأن الحالة قد تتوقف عند Pending Approval قبل Release الفعلي. |

## 9. حالة On Hold

حالة **On Hold** تستخدم لإيقاف أمر Released مؤقتًا عند وجود مشكلة تمنع الاستمرار.

| البند | التوضيح |
|---|---|
| متى تستخدم؟ | عند وجود مشكلة تشغيلية، مواد، موارد، جودة، أو سبب إداري يوقف الأمر. |
| ماذا تمنع؟ | تمنع الحركات على الأمر أثناء الإيقاف. |
| كيف تعود؟ | يمكن إعادتها إلى Released بعد حل المشكلة. |
| أثرها التشغيلي | تمنع استمرار أمر قد ينتج أخطاء أو تكلفة غير صحيحة. |
| المخاطر إذا لم تستخدم | استمرار التنفيذ رغم وجود مشكلة، مما يسبب صرفًا أو إكمالًا أو تكلفة خاطئة. |

---

## 10. Material Issue — صرف المواد

**Material Issue** هو صرف المواد من Supply Subinventory إلى Work Order. هذا الحدث هو أول حدث رئيسي يربط الإنتاج بالمخزون وWIP.

يحدد Oracle سلوك الصرف بناءً على **Supply Type** في Operation Item.

| Supply Type | متى يستخدم؟ | أثره على الصرف | ملاحظات |
|---|---|---|---|
| **Push** | عندما يجب أن يصرف المستخدم المادة يدويًا | صرف صريح من شاشة أو Mobile | مناسب للمواد الحساسة أو التي تحتاج رقابة. |
| **Operation Pull** | عندما تُصرف المادة تلقائيًا عند إكمال العملية | Backflush عند العملية المعنية | يعتمد على دقة العملية والكمية. |
| **Assembly Pull** | عندما تُصرف المادة عند إكمال المنتج النهائي | Backflush عند Product Completion | خطر إذا لم يكن الواقع مطابقًا للتعريف. |
| **Bulk** | مواد للعرض أو الاستهلاك العام | لا تُصرف نظاميًا | يجب فهم أثرها حتى لا تُنسى في التكلفة. |
| **Phantom** | مكون وسيط تُفجر مكوناته داخل الأمر | لا يُنتج بأمر مستقل | يقلل الأوامر لكنه يحتاج تعريفًا دقيقًا. |
| **Supplier** | مواد يقدمها المورد في Outside Processing | لا تُصرف بنفس منطق المواد الداخلية | مرتبطة بعمليات المورد. |

| حالة الصرف | أثرها المحتمل |
|---|---|
| صرف مطابق للمتطلبات | يخفض المخزون ويرفع WIP بالقيمة المناسبة. |
| صرف أقل من المطلوب | قد يؤدي إلى نقص إنتاج أو Material Usage Variance. |
| صرف أكثر من المطلوب | يرفع WIP وتكلفة الأمر، وقد ينتج انحرافًا. |
| صرف مادة بديلة | قد يولد Material Substitution Variance في Standard Cost. |
| صرف Lot منتهي | يمكن منعه بإعداد Prevent Issue of Expired Lots. |
| مادة Supplier Supply | ترتبط بمورد أو Outside Processing لا بصرف داخلي عادي. |
| مكون Phantom | تُضاف مكوناته للأمر بدل إنتاجه كصنف مستقل. |

### أثر Material Issue

| البعد | الأثر |
|---|---|
| المخزون | إنقاص Supply Subinventory بحركة WIP Material Issue. |
| WIP | زيادة WIP بقيمة المواد المصروفة. |
| التكلفة | تنتقل الحركة إلى Cost Accounting عبر عمليات نقل ومعالجة. |
| المحاسبة | تظهر لاحقًا كـ Cost Distribution ثم Subledger Accounting حسب الإعدادات. |
| التتبع | Lot/Serial يجب إدخاله إذا كان الصنف يتطلب ذلك. |

---

## 11. Execute Operations — تنفيذ العمليات

**Operation** هي خطوة منفصلة في تصنيع المنتج. ترتبط بمركز عمل واحد، وقد تكون إلزامية أو تلقائية أو اختيارية.

| نوع العملية | معناها | أثرها على التنفيذ | المخاطر |
|---|---|---|---|
| **Count Point Operation** | عملية يجب تسجيلها صراحة ولا يمكن تجاوزها | تفرض نقطة رقابة في الدورة | عدم تسجيلها يمنع اكتمال صحيح للعملية. |
| **Auto-Transact Operation** | عملية تُسجل تلقائيًا عند إكمال عملية لاحقة | تقلل الإدخال اليدوي | خطر إذا كان الواقع لا يطابق التسلسل. |
| **Optional Operation** | عملية اختيارية يمكن تجاوزها | مرونة تشغيلية | قد لا تُسجل تكلفتها إذا لم تنفذ أو لم تسجل. |

تؤثر العمليات على Product Completion لأن إكمال العملية الأخيرة يؤدي إلى إكمال المنتج، وقد يحول Work Order إلى Completed عند إكمال الكمية الكاملة.

---

## 12. Resource Charging — تسجيل الموارد

**Resource Charging** هو تسجيل استخدام مورد مثل آلة أو عامل على عملية داخل Work Order. في Oracle، تكلفة الموارد تدخل إلى WIP عبر Cost Accounting بناءً على Resource Usage وResource Rate.

| البند | أثره التشغيلي | أثره على التكلفة |
|---|---|---|
| **Manual Resource Charging** | المستخدم يسجل استخدام المورد يدويًا | يعتمد على دقة الإدخال. |
| **Automatic Resource Charging** | النظام يسجل المورد تلقائيًا حسب التعريف | يقلل الإدخال لكنه يحتاج تعريفًا صحيحًا. |
| **Operator Check-in / Check-out** | يربط المشغل أو محطة العمل بالعملية | قد يدعم تتبع وقت أو مورد فعلي. |
| **Resource Usage** | كمية استخدام المورد | تضرب في المعدل لتكوين تكلفة المورد. |
| **Resource Rate** | معدل تكلفة المورد | مصدر Resource Cost في WIP. |

### مخاطر عدم تسجيل الموارد بدقة

- تكلفة تشغيل ناقصة أو مبالغ فيها.
- انحراف Resource Efficiency أو Resource Rate.
- تقارير تكلفة غير موثوقة.
- صعوبة معرفة تكلفة العملية أو مركز العمل.

---

## 13. Material Return — إرجاع المواد

**Material Return** يعكس صرف مادة سبق صرفها إلى Work Order. يحدث عند صرف زائد، مادة خاطئة، مادة معيبة، أو عدم الحاجة للكمية المصروفة.

| الحالة | الأثر المخزني | الأثر على WIP | الأثر المحاسبي |
|---|---|---|---|
| إرجاع مادة زائدة | زيادة Supply Subinventory | خفض WIP | عكس أثر الصرف عبر Cost Accounting. |
| إرجاع مادة خاطئة | تصحيح المخزون | تخفيض تكلفة الأمر | يحتاج تتبع واضح للحركة الأصلية. |
| إرجاع Lot-controlled Material | زيادة نفس Lot أو حسب السياسة | خفض WIP بقيمة الحركة | يجب تحديد Lot الأصلي عند الحاجة. |
| عدم تسجيل المرتجع | المخزون يبقى ناقصًا | WIP يبقى متضخمًا | التكلفة والمالية تصبحان غير دقيقتين. |

---

## 14. Product Completion

**Product Completion** يعني تأكيد إنتاج كمية من المنتج النهائي وإيداعها في المخزون. غالبًا يحدث عند العملية الأخيرة.

| الأثر | ماذا يحدث عند Product Completion؟ |
|---|---|
| التشغيل | تسجيل إنجاز كمية من المنتج النهائي، وقد يكون جزئيًا أو كاملًا. |
| المخزون | زيادة Finished Goods في Completion Subinventory/Locator. |
| WIP | خفض WIP بقيمة الإكمال التقديرية. |
| التكلفة | الإكمال يُسعّر بتكلفة تقديرية Provisional Completion Cost حتى Close. |
| المحاسبة | Cost Accounting يعالج الحركة لاحقًا، والتكلفة الفعلية تُحسب عند الإغلاق. |
| المخاطر | إكمال منتج قبل معالجة المواد أو الموارد أو الجودة يؤدي إلى تكلفة أو مخزون غير دقيق. |

### نقاط مهمة

- في **Discrete Work Definition**، Completion Subinventory/Locator يكون على مستوى رأس Work Definition.
- في **Process Work Definition**، يكون الإكمال على مستوى Operation Output.
- يدعم Oracle الإكمال الجزئي.
- يدعم Undercompletion Tolerance، أي إمكانية اعتبار الأمر مكتملًا رغم إنتاج أقل من الكمية المخططة ضمن سماحية محددة.
- عند إكمال الكمية الكاملة، يتحول Work Order تلقائيًا إلى Completed.

---

## 15. حالة Completed

حالة **Completed** تعني أن العملية الأخيرة اكتملت وتم إكمال الكمية المطلوبة أو ضمن سماحية مقبولة. لكنها ليست نهاية الدورة المالية.

| Completed | Closed |
|---|---|
| تعني أن التنفيذ اكتمل تشغيليًا. | تعني أن الأمر أُغلق ماليًا وتشغيليًا. |
| قد تبقى التكلفة تقديرية. | يتم حسم Actual Cost أو Variances. |
| يمكن أن توجد مراجعات قبل الإغلاق. | لا يسمح بالتعديل بعد الإغلاق. |
| WIP قد يحتاج تسوية. | WIP يجب أن يُسوّى. |
| ليست كافية للإقفال المحاسبي. | مرتبطة مباشرة بالـ Cost Accounting وPeriod Close. |

قبل Close يجب مراجعة المواد، الموارد، المرتجعات، الهالك، الجودة، والحركات المعلقة.

---

## 16. Work Order Close

**Work Order Close** هو أهم حدث محاسبي في دورة Oracle Manufacturing. ليس مجرد تغيير حالة؛ بل حدث تسوية تكلفة، WIP، وانحرافات.

عند الإغلاق، يتأكد Cost Processor أن جميع الحركات الخاصة بالأمر قد عولجت بنجاح. ثم تظهر الانحرافات أو تعديلات التكلفة حسب Cost Method.

### ماذا يحدث عند Close؟

- التأكد من عدم وجود Pending Transactions.
- التحقق من اكتمال العمليات اللازمة.
- تسوية WIP.
- حساب التكلفة الفعلية في Actual / Average.
- توليد Variances في Standard Cost.
- إعادة حساب Scrap عند الحاجة.
- منع التعديل بعد Closed.
- ربط الإغلاق بالفترة المحاسبية الصحيحة.

### Checklist قبل إغلاق Work Order

| بند التحقق | لماذا مهم؟ | الخطر إذا لم يتحقق |
|---|---|---|
| المواد المصروفة | لضمان أن تكلفة المواد مكتملة | تكلفة ناقصة أو WIP غير صحيح. |
| المرتجعات | لإزالة المواد الزائدة من WIP | تضخم تكلفة الأمر. |
| الموارد المسجلة | لضمان إدخال تكلفة العمالة/الآلات | تكلفة تشغيل ناقصة أو Variances غير مفسرة. |
| العمليات المنجزة | لأن Count Point يجب أن يكتمل | إغلاق أمر لم ينفذ فعليًا. |
| المنتج المكتمل | للتأكد من أن Product Completion صحيح | Finished Goods غير مطابق. |
| الجودة | لمنع إغلاق أمر بمنتج مرفوض أو تحت فحص | مخزون قابل للبيع رغم مشكلة جودة. |
| الهالك | لضمان Scrap Accounting الصحيح | تكلفة المنتج أو مصروف الهالك غير صحيح. |
| الحركات المعلقة | Oracle لا يغلق إذا توجد حركات معلقة | فشل الإغلاق وتأخير Period Close. |
| WIP | للتأكد من التسوية | WIP مفتوح أو متضخم. |
| التكلفة | لضمان معالجة Cost Accounting | تكلفة غير نهائية. |
| الفترة المحاسبية | لتجنب انتقال فروقات لفترة خاطئة | هامش ربح غير دقيق. |
| الصلاحيات | لأن Close حدث حوكمة | إغلاق غير مصرح أو خاطئ. |

### لماذا تأخير الإغلاق خطر؟

إذا لم تُغلق أوامر العمل، قد تظهر Product Cost Adjustments أو Variances في فترة مختلفة عن فترة Product Completion، مما يؤدي إلى تشويه هامش الربح والتقارير المالية.

---

## 17. حالة Closed

| البند | التوضيح |
|---|---|
| المعنى | إغلاق نهائي لأمر العمل بعد معالجة التكلفة والانحرافات. |
| ما يمنع بعدها | لا يسمح بالتعديل العادي أو الحركات التنفيذية. |
| علاقتها بالانحرافات | تظهر Variances أو Cost Adjustments عند الإغلاق حسب منهج التكلفة. |
| علاقتها بالمحاسبة | تُبنى Cost Distributions وتنتقل لاحقًا إلى Subledger ثم GL. |
| لماذا هي نقطة حوكمة؟ | لأنها تقفل دورة التنفيذ والتكلفة وتمنع التلاعب اللاحق. |
| خطر إغلاق أمر خاطئ | تكلفة غير صحيحة، WIP غير متوازن، وانحرافات مضللة. |

Oracle يدعم إعادة فتح الأمر المغلق ثم إغلاقه مجددًا، لكن ذلك يحتاج حوكمة واضحة. التفاصيل العملية الدقيقة تحتاج تحقق إضافي من المصدر الأصلي.

---

## 18. حالة Canceled

حالة **Canceled** تعني إلغاء أمر العمل. حسب المقال، يمكن الانتقال إليها من Unreleased أو Released، وبعدها لا يسمح بالتعديل.

| البند | التوضيح |
|---|---|
| متى تستخدم؟ | عندما لا يعود الأمر مطلوبًا أو يجب إيقافه نهائيًا. |
| هل يمكن إلغاء أمر عليه حركات؟ | يحتاج تحقق إضافي من المصدر الأصلي. |
| ما المخاطر؟ | إلغاء أمر له مواد مصروفة أو حجوزات أو ارتباطات قد يسبب أثرًا مخزنيًا أو ماليًا غير واضح. |
| ما يجب مراجعته؟ | الحركات، المواد المصروفة، الحجوزات، الطلبات المرتبطة، والتكلفة. |
| أثره على الطلبات أو الحجوزات | يحتاج تحقق إضافي من المصدر الأصلي. |

---

## 19. Production Exceptions / Quality / Maintenance

تدخل الاستثناءات والجودة والصيانة في الدورة عندما تظهر مشاكل في أرضية المصنع أو في الموارد أو المواد أو الجودة.

| الاستثناء | متى يظهر؟ | أثره | الإجراء |
|---|---|---|---|
| **Production Exception** | عند مشكلة تؤثر على تقدم Work Order | قد يوقف أو يعرقل التنفيذ | تسجيل الاستثناء وربطه بالأمر أو المورد أو المادة. |
| **Resource Exception** | عند عطل آلة أو غياب عامل | يؤثر على تنفيذ العملية والجدولة | متابعة المورد أو إنشاء إجراء صيانة. |
| **Material Exception** | عند نقص مادة أو مشكلة مكون | يمنع أو يؤخر الصرف والتنفيذ | معالجة التوريد أو البدائل. |
| **Quality Issue** | عند مشكلة جودة | قد تمنع الإتاحة أو تتطلب Rework/Scrap | تسجيل فحص أو رفض أو قرار معالجة. |
| **Inspection** | عند عملية فحص أو Quality Check | يقرر قبول أو رفض المنتج | ربط الفحص بالعملية أو المنتج. |
| **Maintenance Integration** | عند مشكلة آلة أو مورد | قد تنتج أوامر صيانة أو استثناءات موارد | التنسيق بين الإنتاج والصيانة. |
| **Rework** | عند منتج قابل للإصلاح | يفتح دورة إعادة تشغيل | Rework Work Order عند الحاجة. |
| **Reject** | عند رفض كمية | يؤثر على المخزون والجودة | Scrap أو Rework حسب القرار. |
| **Scrap** | عند تلف كمية لا يمكن إصلاحها | يخفض WIP أو يحمل على المنتج/المصروف | تسجيل Scrap Transaction وسياسة تكلفة. |

## 21. أهم المخاطر في دورة Work Order

| الخطر | أثره | كيف نتحكم به؟ |
|---|---|---|
| Work Definition خاطئة | أمر عمل خاطئ من البداية | اعتماد Work Definition ومراجعة الإصدارات. |
| Work Definition Version غير مناسبة | استخدام طريقة إنتاج غير صالحة زمنيًا | ضبط Effective Dates. |
| Supply Type غير صحيح | صرف غير مناسب أو Backflush خاطئ | مراجعة Operation Items قبل Go-Live. |
| صرف مواد خاطئ | مخزون وWIP غير دقيقين | رقابة Material Issue وتقارير المقارنة. |
| عدم تسجيل الموارد | تكلفة ناقصة | ضبط Resource Charging. |
| عدم تسجيل المرتجعات | WIP متضخم ومخزون ناقص | سياسة Material Return واضحة. |
| Product Completion قبل اكتمال البيانات | Finished Goods وتكلفة غير دقيقة | Validations قبل الإكمال. |
| حركات معلقة تمنع الإغلاق | فشل Close وPeriod Close | تقرير Pending Transactions. |
| تأخير Work Order Close | فروقات في فترة محاسبية خاطئة | سياسة إغلاق دورية. |
| WIP غير دقيق | ميزانية وتكلفة غير صحيحة | WIP Report ومراجعة أوامر مفتوحة. |
| Cost Distributions غير مراجعة | فجوة بين الإنتاج والمالية | مراجعة Cost Accounting. |
| Period Close قبل تسوية أوامر الإنتاج | هامش ربح غير دقيق | Period Close Validation. |
| Lot / Serial غير مضبوط | ضعف التتبع والجودة | تفعيل Lot/Serial عند الحاجة. |
| عدم استخدام On Hold عند وجود مشكلة | استمرار أخطاء التنفيذ | تدريب المشرفين على On Hold. |

## 23. دروس مستفادة لنظام ERP محلي مثل ناتج

- ضرورة وجود **Work Order Status Workflow** واضح، وليس أمر إنتاج مفتوح بلا حالات.
- أهمية فصل **تعريف طريقة الإنتاج** عن **أمر الإنتاج التنفيذي**.
- الحالات الأساسية مثل Unreleased وReleased وCompleted وClosed تعطي حوكمة حقيقية.
- Supply Type ليس تفصيلًا ثانويًا؛ هو الذي يحدد متى وكيف تُصرف المواد.
- Material Issue يجب أن يرتبط بالمخزون وWIP وليس مجرد سطر إدخال.
- Product Completion يجب أن يضيف المنتج النهائي ويؤثر على WIP والتكلفة.
- Work Order Close يجب أن يكون حدثًا ماليًا، لا زر إغلاق شكلي.
- بعد Close يجب منع التعديل العادي أو ضبطه بصلاحيات صارمة.
- تقارير أوامر العمل المفتوحة وPending Transactions ضرورية.
- WIP وCost Distributions يجب أن تكون قابلة للمراجعة، خصوصًا عند العملاء الكبار.

## 24. ملخص تنفيذي

دورة Work Order في Oracle تبدأ من **Work Definition** وليس من أمر العمل فقط. فقبل أن تنشئ أمرًا، يجب أن تكون الأصناف، Item Structure، Work Centers، Resources، Operations، وPlant Parameters مضبوطة.

حالة **Unreleased** تسمح بالمراجعة والتعديل ولا تسمح بالتنفيذ. عند **Released** يصبح الأمر جاهزًا لصرف المواد وتنفيذ العمليات. **Material Issue** يخفض مخزون المواد ويرفع WIP. **Resource Charging** يضيف تكلفة العمالة أو الآلات إلى WIP. **Product Completion** يدخل المنتج النهائي إلى المخزون بتكلفة تقديرية. بعد ذلك، لا تكون الدورة منتهية ماليًا إلا عند **Work Order Close**.

الإغلاق هو لحظة حسم التكلفة: في Standard Cost تظهر الانحرافات، وفي Actual / Average تظهر تعديلات تكلفة المنتج. كما تتم تسوية WIP وربط النتائج بالفترة المحاسبية الصحيحة. لذلك، تأخير إغلاق أوامر العمل يؤدي إلى تكلفة تظهر في فترة غير مناسبة، وقد يشوه الربحية والتقارير المالية.

نجاح دورة Oracle يعتمد على ثلاثة عناصر حاسمة: **Work Definition صحيحة، Supply Types واضحة، وإغلاق منتظم لأوامر العمل**. بدون هذه العناصر، ستعمل الشاشة ظاهريًا، لكن المخزون والتكلفة وWIP والتقارير لن تكون موثوقة.
