# الدرس 03: الكيانات الرئيسية في موديول الإنتاج

## لماذا هذا الدرس مهم

الكثير من المبتدئين يفهم الإنتاج كقائمة شاشات.

لكن الفهم الصحيح يبدأ من فهم الكيانات Entities التي يتكون منها الموديول.

عندما تفهم الكيانات الأساسية، ستفهم:

- ما الذي يجب أن يوجد في قاعدة البيانات أو منطق النظام.
- ما الذي يجب أن يكون معرفًا قبل التشغيل.
- ما الذي يتحرك أثناء التنفيذ.
- أين يمكن أن تظهر الفجوات في النظام المحلي.

## الهدف من الدرس

بعد إنهاء هذا الدرس يجب أن تكون قادرًا على:

1. تسمية الكيانات الأساسية في أي Production Module.
2. فهم علاقة هذه الكيانات ببعضها.
3. التمييز بين كيان تعريف Design Entity وكيان تنفيذ Execution Entity.
4. قراءة أي نظام محلي بعقلية Entity-Based Thinking.

## ما معنى الكيان Entity هنا؟

الكيان Entity هو شيء مستقل منطقيًا داخل النظام، له بياناته، وعلاقاته، وسلوكه.

مثلًا:

- Product كيان
- BOM كيان
- Route كيان
- Production Order كيان
- Inventory Transaction كيان

المستشار القوي لا يرى فقط الشاشة.

بل يرى الكيانات التي يجب أن تكون خلف الشاشة.

## الكيانات الأساسية الثمانية

### 1. كيان المنتج Product Entity

هذا الكيان يمثل ما سيتم إنتاجه أو استهلاكه.

قد يظهر كالتالي:

- Released Product in Dynamics
- Item in Oracle
- Material Master in SAP

البيانات المهمة هنا تشمل:

- الكود Code
- الاسم Name
- النوع Type
- وحدة القياس UoM
- التتبع Tracking
- الموقع Site / Plant / Organization

بدون Product Entity مضبوط، لا معنى لبقية الدورة.

### 2. كيان بنية المنتج Product Structure Entity

هذا هو الكيان الذي يحدد مكونات المنتج.

قد يظهر كالتالي:

- BOM in Dynamics and SAP
- Item Structure or BOM in Oracle

هذا الكيان يحتوي عادةً على:

- Components
- Quantities
- UoM
- Scrap
- Alternatives or Versions

هذا الكيان يخدم التخطيط والاستهلاك والتكلفة معًا.

### 3. كيان طريقة التصنيع Production Method Entity

هذا يشرح كيف يُصنع المنتج، وليس مم يتكون.

قد يظهر كالتالي:

- Route / Routing
- Operations
- Resources
- Work Definition in Oracle as an integrated entity

البيانات المهمة:

- Operations sequence
- Work Centers
- Resources
- Time standards
- Cost drivers

### 4. كيان التنفيذ التنفيذي Order Entity

هذا هو الكيان الذي يمثل الحدث التنفيذي الفعلي.

قد يظهر كالتالي:

- Production Order
- Work Order
- Process Order

هذا الكيان يجمع في لحظة التنفيذ:

- Product
- Quantity
- Structure snapshot
- Method snapshot
- Status
- Transactions
- Cost accumulation

هذا الكيان هو قلب التنفيذ.

### 5. كيان الحركة المخزنية Inventory Transaction Entity

أي Production Module قوي لا يعيش من دون Inventory Transactions واضحة.

مثل:

- Material Issue / Goods Issue
- Finished Goods Receipt / Product Completion / Report as Finished
- Material Return

هذه الكيانات الفرعية أو السجلات تعطيك إجابة عن:

- ما الذي خرج من المخزون؟
- ما الذي دخل؟
- متى؟
- على أي أمر؟

### 6. كيان الحركة التشغيلية Execution Transaction Entity

وهذا يخص ما يحدث في التشغيل نفسه، مثل:

- Operation Execution
- Confirmation
- Resource Charging
- Job Reporting
- Route Reporting

هذا الكيان مهم لأنه يمثل العمل الفعلي على الأرض، وليس فقط حركة المواد.

### 7. كيان التكلفة Cost Entity

حتى لو لم يظهر دائمًا كشاشة مستقلة، إلا أن منطق التكلفة يجب أن يكون موجودًا.

مثل:

- Estimated Cost
- Actual Cost
- WIP
- Variance
- Cost Distribution
- Settlement

إذا لم تجد Cost Entity أو Cost Logic واضحة في النظام، فاعرف أن الموديول ناقص من الناحية الرقابية أو المالية.

### 8. كيان الجودة والرقابة Quality and Control Entity

هذا الكيان قد يظهر عبر:

- Quality Order
- Quality Issue
- Inspection Record
- Scrap Record
- Rework Logic
- Control Reports

الجودة ليست شيئًا إضافيًا دائمًا، بل جزء من نضج الموديول.

## كيف ترتبط هذه الكيانات ببعضها

العلاقة المنطقية الأساسية تكون غالبًا كالتالي:

Product
→ Product Structure
→ Production Method
→ Production Order / Work Order
→ Inventory Transactions
→ Execution Transactions
→ Cost Accumulation
→ Quality and Control
→ Closing and Reporting

وهذا يعني:

- المنتج Product هو الأصل.
- BOM أو Structure تشرح ماذا يدخل.
- Route أو Method تشرح كيف يُصنع.
- Order يحول التعريف إلى تنفيذ.
- Transactions تحوّل التنفيذ إلى أثر حقيقي.
- Cost and Quality يفسران أثر التنفيذ ويراقبانه.

## كيف يظهر هذا في الأنظمة الثلاثة

### Dynamics 365

يميل إلى فصل الكيانات بوضوح:

- Released Product
- BOM / Formula
- Route
- Production Order
- Picking List Journal
- Route / Job Card Journals
- Report as Finished
- Costing / Variance / End

### Oracle Fusion Cloud

يميل إلى دمج جزء من تعريف التصنيع في Work Definition:

- Item
- Item Structure
- Work Definition
- Work Order
- Material Issue
- Resource Charging
- Product Completion
- Cost Distribution
- Work Order Close

### SAP S/4HANA

يميل إلى عمق أكبر في Master Data and Cost Control:

- Material Master
- BOM
- Routing
- Production Version
- Production Order / Process Order
- Goods Issue / Goods Receipt / Confirmation
- WIP / Variance / Settlement

## منظور المستشار Functional Consultant View

عندما تحلل نظامًا محليًا، اسأل:

- هل هذه الكيانات موجودة أصلًا؟
- هل هي كيانات مستقلة فعلًا أم منطقها مفقود؟
- هل الربط بينها واضح؟
- هل بعضها موجود اسميًا فقط دون أثر حقيقي؟

مثال:

قد تجد شاشة باسم Production Order، لكن:

- لا توجد BOM حقيقية
- لا توجد حالات تنفيذ
- لا توجد Transactions واضحة
- لا يوجد WIP

إذن الكيان هنا موجود بالاسم فقط، لكنه ضعيف وظيفيًا.

## الأخطاء الشائعة Common Mistakes

1. الخلط بين الكيانات التعريفية Design Entities والكيانات التنفيذية Execution Entities.
2. اعتبار أمر الإنتاج Production Order هو كل الموديول.
3. نسيان أن Inventory Transactions وCost Logic كيانات حقيقية في منطق العمل.
4. إهمال Quality Entity عند تقييم نضج الموديول.

## المراجعة السريعة Quick Review

- Product هو الأصل.
- Product Structure يشرح ماذا يدخل.
- Production Method يشرح كيف يُصنع.
- Order هو الكيان التنفيذي المركزي.
- Inventory Transactions تعكس حركة المواد والمنتج.
- Execution Transactions تعكس العمل الفعلي.
- Cost Entity تفسر الأثر المالي.
- Quality and Control Entity ترفع نضج الموديول.
