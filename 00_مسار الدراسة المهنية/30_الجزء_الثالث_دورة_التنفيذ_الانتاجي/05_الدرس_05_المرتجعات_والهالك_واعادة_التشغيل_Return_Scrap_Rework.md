# الدرس 05: المرتجعات والهالك وإعادة التشغيل Return Scrap Rework

## لماذا هذا الدرس مهم

لأن أي دورة تنفيذ Production Execution Cycle مثالية على الورق فقط. في الواقع العملي تظهر مواد مصروفة زيادة ثم تعاد، وقطع تالفة Scrap، ومنتجات تحتاج Rework أو Repair. إذا لم تفهم هذه السيناريوهات، فسيبدو لك النظام جيدًا فقط عندما يكون الواقع مثاليًا، وهذا لا يحدث في المصانع الحقيقية.

## الهدف من الدرس

بعد هذا الدرس يجب أن تكون قادرًا على:

- شرح الفرق بين Material Return وScrap وRework.
- فهم أثر كل حالة على المخزون والتكلفة.
- تفسير لماذا يعد الهالك جزءًا من التحكم وليس مجرد خسارة.
- صياغة أسئلة استشارية صحيحة عن سياسات العيوب والانحراف.

## الشرح المفاهيمي الأساسي

### Material Return

Material Return هو عكس صرف تم سابقًا على أمر الإنتاج.

يستخدم عندما:

- صرفت كمية أكبر من المطلوب.
- ألغيت عملية أو خفضت كمية الأمر.
- أعيدت مادة صالحة إلى المخزون.

### Scrap

Scrap هو فقد فعلي في المواد أو المنتج أو المخرجات لا يُتوقع أن يعود كمخزون صالح للاستخدام المباشر.

قد يظهر Scrap في:

- المواد الخام Raw material scrap
- أثناء العملية Operation scrap
- المنتج النهائي Finished goods scrap

### Rework

Rework يعني أن الناتج غير مطابق من أول مرة، لكنه لا يزال قابلًا للإصلاح أو إعادة التشغيل بدل إعدامه بالكامل.

وهنا تظهر أسئلة مهمة:

- هل Rework يتم داخل نفس الأمر أم بأمر مستقل؟
- هل يحمل تكلفة مستقلة؟
- هل نحتاج سبب عيب Defect reason أو Nonconformance code؟

## الفروق الجوهرية

| الحالة | هل تعود للمخزون؟ | هل تؤثر على التكلفة؟ | هل تحتاج تتبع سبب؟ |
| --- | --- | --- | --- |
| Material Return | نعم غالبًا | نعم | أحيانًا |
| Scrap | لا في المعتاد | نعم بقوة | نعم |
| Rework | ليس مباشرة | نعم | نعم |

## كيف يظهر المفهوم في الأنظمة الثلاثة

### Dynamics 365

في Dynamics يمكن التعامل مع Scrap عبر الكميات التالفة أو الانحرافات أو الأوامر المرتبطة بالجودة أو المسارات التشغيلية. كما يمكن عكس بعض حركات المواد وفق نوع الجورنال Journal المنفذ. أما إعادة التشغيل فقد تظهر كجزء من مسار تشغيلي أو كمعالجة عملية مستقلة حسب تصميم العميل.

### Oracle Fusion Cloud

في Oracle يظهر Material Return كحركة مستقلة تعيد المادة إلى المخزون وتخفض أثرها على Work Order. أما Scrap وRejects فقد ترتبط بالتنفيذ وبالجودة وبالمخرجات المقبولة والمرفوضة. Rework قد يحتاج تصميمًا وظيفيًا دقيقًا بحسب ما إذا كان العميل يعيد المعالجة داخل نفس Work Order أو عبر تدفق منفصل.

### SAP S/4HANA

في SAP يظهر Scrap بوضوح داخل Confirmation أو Goods Movements أو عبر إعدادات BOM/Routing مثل Component Scrap وOperation Scrap. أما Rework فقد يظهر عبر Rework Order أو منطق خاص حسب السيناريو. وMaterial Return يمكن تمثيله بحركات عكس أو Return movements مرتبطة بالأمر.

## منظور المستشار Functional Consultant View

عندما يناقش العميل الجودة أو العيوب، لا تكتفِ بالسؤال: هل يوجد Scrap؟

بل اسأل:

- هل Scrap يظهر على مستوى المواد أم العمليات أم المنتج النهائي؟
- هل تريدون تسجيل سبب العيب Defect reason؟
- هل المواد المرتجعة تعود إلى Available stock أم إلى فحص أو حجر؟
- هل Rework يتم داخل نفس الأمر أم بأمر مستقل يسهل قياسه؟
- هل تريدون رؤية تكلفة Rework مستقلة عن تكلفة الإنتاج الأساسي؟

هذه الأسئلة تحمي المشروع من تصميم يبدو بسيطًا لكنه يضيع معلومات حاسمة عن الجودة والتحسين المستمر.

## الأخطاء الشائعة Common Mistakes

- تسجيل Scrap كفرق كمي فقط من دون سبب.
- إعادة المواد إلى المخزون من دون تحقق من حالتها الفعلية.
- دمج Rework داخل التشغيل العادي بحيث تختفي تكلفته.
- تجاهل العلاقة بين الهالك والجودة والتحسين المستمر Continuous improvement.
- بناء تقارير تكلفة دون عزل أثر الخسائر والمرتجعات.

## المراجعة السريعة Quick Review

- Material Return يعكس صرفًا سابقًا.
- Scrap يمثل فقدًا حقيقيًا يجب تتبعه لا إخفاؤه.
- Rework ليس مجرد تصحيح تشغيلي، بل حدث له أثر على الوقت والطاقة والتكلفة.
- التصميم الجيد يربط هذه الحالات بالجودة وبأسباب العيوب.
