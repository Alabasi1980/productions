# قالب مقارنة نظام الإنتاج في Oracle Fusion Cloud Manufacturing مع نظام ERP محلي مثل ناتج

## 1. مقدمة

هذا الملف ليس حكمًا على نظام ناتج، ولا يفترض أن ناتج يدعم أو لا يدعم أي ميزة. هو قالب مقارنة مستخرج من دراسة Oracle Fusion Cloud Manufacturing، هدفه مساعدة مستشار ERP على تقييم Module الإنتاج في نظام ERP محلي بطريقة منظمة.

المقارنة لا تهدف إلى نسخ Oracle حرفيًا، بل إلى:

- فهم الفجوات.
- تحديد الحد الأدنى المطلوب.
- تحديد الميزات المتقدمة.
- ترتيب الأولويات.
- دعم تحسين النظام.
- تحسين تطبيق النظام عند العملاء.
- حماية العلاقة بين الإنتاج، المخزون، التكلفة، والمالية.

---

## 2. طريقة استخدام القالب

يُستخدم هذا الملف لاحقًا عند دراسة Module الإنتاج في نظام ناتج عمليًا، وليس قبل ذلك. طريقة الاستخدام المقترحة:

1. دراسة Module الإنتاج في نظام ناتج عمليًا.
2. ملء عمود **كيف يعالج ناتج هذا المحور؟**.
3. تحديد هل توجد Gap.
4. تصنيف الفجوة.
5. تحديد الأولوية.
6. تحديد هل الحل مطلوب الآن أم لاحقًا.
7. تحويل الفجوات إلى خطة تحسين مرحلية.

| درجة الفجوة | المعنى | الإجراء المقترح |
|---|---|---|
| لا توجد فجوة | النظام يدعم المفهوم أو يعالجه بطريقة بديلة مقبولة | توثيق طريقة الاستخدام وتثبيت التدريب |
| فجوة تدريب | الميزة موجودة لكن المستخدمين لا يفهمونها | تدريب + دليل استخدام + أمثلة تشغيلية |
| فجوة إعدادات | الميزة موجودة لكنها غير مفعلة أو غير مضبوطة | مراجعة الإعدادات وتوثيقها |
| فجوة توثيق | النظام يعمل لكن لا توجد سياسة واضحة | إعداد سياسة تشغيلية أو مالية |
| فجوة تقرير | البيانات موجودة لكن لا يوجد تقرير رقابي | تصميم تقرير أو Dashboard |
| فجوة Workflow | التسلسل الحالي لا يمنع الأخطاء | تعديل حالات أو سير العمل |
| فجوة مالية | الأثر على WIP أو التكلفة أو GL غير واضح | إشراك المالية وتصميم معالجة محاسبية |
| فجوة تطوير بسيطة | تعديل محدود في شاشة أو تحقق أو تقرير | إدراجها ضمن تحسينات قصيرة المدى |
| فجوة تطوير جوهرية | ميزة أساسية غير موجودة | مشروع تطوير مرحلي مستقل |

### الحد الأدنى المطلوب من Item Structure / BOM في نظام ERP محلي

الحد الأدنى الذي يجب التحقق منه في ناتج:

- تعريف مكونات المنتج.
- تحديد الكميات.
- دعم وحدات القياس والتحويلات.
- دعم المنتجات نصف المصنعة إذا كان العميل يحتاج ذلك.
- اعتماد BOM قبل استخدامها.
- ربط BOM بأمر الإنتاج.
- ربط BOM بصرف المواد.
- ربط BOM بتكلفة المنتج على الأقل بمستوى مبدئي.
- منع استخدام BOM غير معتمدة أو غير مكتملة في أوامر الإنتاج.

### هل يحتاج نظام ناتج إلى مفهوم Work Definition مستقل؟

الإجابة يجب أن تكون محايدة ومبنية على حجم عملاء ناتج:

- إذا كان ناتج يخدم عملاء إنتاج بسيط، فقد تكفي **BOM + مراحل إنتاج بسيطة** كبداية.
- إذا كان يخدم مصانع متوسطة أو معقدة، يصبح وجود مفهوم شبيه بـ **Work Definition** مهمًا لأنه يفصل بين طريقة إنتاج المنتج وأمر الإنتاج الفعلي.
- لا يجب نسخ Oracle حرفيًا، لكن يجب فهم قيمة الفصل بين **طريقة الإنتاج** و **أمر الإنتاج**.
- الفجوة هنا قد تكون تدريبًا أو إعدادات أو تطويرًا حسب ما يدعمه ناتج فعليًا.

### هل يحتاج نظام ناتج إلى حالات واضحة لأمر الإنتاج؟

نعم، يجب التحقق من ذلك بقوة. الحالات ليست شكلية، بل تمنع:

- تنفيذ حركات في وقت غير مناسب.
- صرف مواد قبل جاهزية الأمر.
- استلام منتج قبل بدء الإنتاج.
- تعديل أمر بعد الإغلاق.
- إغلاق مالي غير صحيح.
- بقاء WIP مفتوحًا بلا رقابة.
- عدم وضوح المسؤولية بين الإنتاج والمستودعات والمالية.

- صرف يدوي.

- دعم مواد من مورد خارجي عند وجود عمليات خارجية.
- تحديد مستودع الصرف لكل مادة أو لكل أمر.

### الفرق بين Product Completion وWork Order Close

في Oracle، قد يدخل المنتج النهائي إلى المخزون عند **Product Completion** قبل أن تكون التكلفة نهائية. التكلفة النهائية تُحسم عند **Work Order Close**. عند مقارنة ناتج، يجب التحقق هل يخلط النظام بين الاستلام التشغيلي والإغلاق المالي، أم يفصل بينهما.

- دعم الرفض أو إعادة التشغيل عند الحاجة.
- تقرير هالك حسب أمر/منتج/فترة.

### هل يمكن لنظام إنتاج أن ينجح بدون WIP واضح؟

- في العملاء الصغار قد يبدأ النظام بتكلفة مبسطة دون WIP كامل، بشرط أن يكون ذلك مفهومًا ومقبولًا من المالية.

- يجب على ناتج تحديد مستوى WIP المناسب حسب نوع العميل، وليس نسخ Oracle بالكامل أو تجاهله بالكامل.

### أسئلة حرجة يجب طرحها على نظام ناتج بخصوص التكلفة

1. هل يحسب تكلفة أمر إنتاج أم تكلفة منتج فقط؟
2. هل تكلفة المنتج تأتي من BOM فقط أم تشمل الموارد؟
3. هل يوجد فرق بين التكلفة التقديرية والفعلية؟
4. هل يوجد إغلاق أمر إنتاج يحسم التكلفة؟
5. هل يوجد أثر WIP؟
6. هل توجد تقارير تكلفة حسب أمر الإنتاج؟
7. هل يوجد تحليل فروقات؟
8. هل يمكن ربط التكلفة بالمحاسبة؟
9. هل يدعم أكثر من طريقة تكلفة؟
10. هل المالية توافق على طريقة التكلفة الحالية؟

### هل يحتاج نظام ناتج إلى Cost Distributions مثل Oracle؟

لا يجب نسخ Oracle كاملًا، لكن يجب وجود سجل واضح يربط كل حركة إنتاج بأثرها المالي أو التكلفي. في الأنظمة البسيطة يمكن البدء بتقارير تكلفة واضحة قبل القيود التلقائية الكاملة. أما في العملاء المتوسطين والكبار، فإن غياب سجل ربط بين الإنتاج والمالية سيجعل المطابقة والمراجعة صعبة.

## 28. طريقة تحويل الفجوات إلى خطة تحسين

بعد تعبئة نموذج Gap Analysis، لا يتم تحويل كل فجوة مباشرة إلى تطوير. الخطوات الصحيحة:

1. تجميع الفجوات المتشابهة.
2. حذف الفجوات غير المهمة لعملاء ناتج الحاليين.
3. تحديد الحد الأدنى اللازم لتشغيل إنتاج موثوق.
4. فصل التدريب عن الإعدادات.
5. فصل التقارير عن تعديلات Workflow.
6. فصل التكامل المالي عن التحسينات التشغيلية.
7. ترتيب الأولويات.
8. كتابة Phase 1 / Phase 2 / Phase 3.
9. اعتماد الخطة مع الإدارة أو فريق التطوير.

| نوع الفجوة | مثال | طريقة المعالجة |
|---|---|---|
| تدريب | المستخدم لا يعرف الفرق بين Completed وClosed | جلسة تدريب + دليل تشغيل |
| إعدادات | مستودع الصرف غير محدد | ضبط إعدادات المستودعات |
| توثيق | لا توجد سياسة إغلاق أمر إنتاج | إعداد SOP |
| تقرير | لا يوجد تقرير أوامر غير مغلقة | تصميم تقرير |
| Workflow | النظام يسمح بالإغلاق قبل مراجعة المواد | تعديل سير الحالة |
| صلاحيات | أي مستخدم يستطيع تعديل BOM | ضبط أدوار وصلاحيات |
| مالية | لا يوجد أثر WIP واضح | تصميم معالجة تكلفة |
| تطوير بسيط | إضافة حقل سبب الهالك | تعديل شاشة |
| تطوير جوهري | إضافة مفهوم Work Definition أو WIP كامل | مشروع تطوير مرحلي |
| تكامل محاسبي | لا توجد مطابقة بين الإنتاج والمالية | تصميم تكامل مالي |

## 29. تحذيرات مهمة عند المقارنة

- لا يعني أن Oracle يدعم ميزة أن ناتج يجب أن يدعمها فورًا.
- لا تنقل تعقيد Oracle إلى عملاء صغار.
- ركّز على القيمة العملية لا على تقليد النظام.
- بعض المفاهيم متقدمة ولا تصلح لكل عميل.
- يجب التفريق بين نقص النظام ونقص تدريب العميل.
- يجب التفريق بين Gap حقيقي وFeature غير ضرورية.
- يجب إشراك المالية عند مقارنة التكلفة وWIP وClose.
- يجب عدم طرح فجوات تطوير كبيرة قبل فهم نظام ناتج بالكامل.
- يجب التحقق هل المشكلة في النظام أم في الإعدادات أم في الاستخدام.
- لا تجعل المقارنة مع Oracle سببًا لتعقيد نظام محلي فوق حاجة السوق.
- الأولوية دائمًا: تشغيل صحيح، مخزون صحيح، تكلفة مفهومة، تقارير رقابية.

## 30. ملخص تنفيذي

هذا الملف أداة مقارنة وليس حكمًا على نظام ناتج.

الهدف هو فهم أين يقف نظام ناتج مقابل مفاهيم Oracle Manufacturing، لكن الأهم ليس نسخ Oracle، بل تحديد ما هو أساسي لعملاء ناتج وما يمكن تأجيله. المقارنة يجب أن تنتج خطة تحسين عملية، لا قائمة أمنيات.

أكثر المحاور حساسية عند المقارنة:

- Work Definition أو ما يعادله.
- Work Order Lifecycle وحالات الأمر.
- Material Issue وربطه بالمخزون.
- Product Completion وربطه باستلام المنتج.
- WIP.
- Cost Accounting أو بديله المحلي.
- Work Order Close.
- Reports.
- UAT وGo-Live.

أفضل نتيجة من هذا الملف هي تحويل الفجوات إلى قرارات واضحة:

- تدريب.
- إعدادات.
- توثيق.
- تقارير.
- تحسين Workflow.
- صلاحيات.
- تكامل مالي.
- تطوير مرحلي.

لا تبدأ بخطة تطوير قبل التأكد أن الفجوة ليست تدريبًا أو إعدادًا أو سوء استخدام.
