# نموذج الإنتاج العملي في Odoo Manufacturing

## 1. مقدمة

هذا الملف يقدم نموذجًا تدريبيًا عمليًا لفهم **Odoo Manufacturing** من زاوية مستشار ERP يريد فهم الإنتاج والتصنيع بطريقة قابلة للتطبيق، لا بطريقة نظرية عميقة مثل SAP أو Oracle.

Odoo ليس المرجع الأعمق في موضوعات مثل **WIP العميق، Variance الرسمي، Settlement، Cost Object Controlling**، لكنه مرجع ممتاز لفهم كيف يمكن تحويل الإنتاج إلى نظام عملي واضح مناسب للشركات الصغيرة والمتوسطة والـMid-Market.

قيمة Odoo هنا تظهر في:

- تبسيط دورة الإنتاج.
- وضوح تجربة المستخدم.
- ربط **BOM** بأمر التصنيع.
- ربط **Work Orders** بمراكز العمل.
- ربط **Shop Floor** بالتنفيذ اليومي.
- ربط الجودة والصيانة والهالك بالتشغيل.
- عرض **MO Cost vs Real Cost** على مستوى Manufacturing Order.
- تقديم نموذج مفهوم للمبرمجين عند بناء Module إنتاج محلي.

الفكرة الأساسية:  
Odoo يساعدنا على رؤية الإنتاج كعملية عملية قابلة للاستخدام، وليس كنموذج محاسبي معقد.

## 2. الفكرة العامة لنموذج Odoo Manufacturing

النموذج العملي في Odoo يدور حول عناصر مترابطة:

- **Product**
- **BOM**
- **Manufacturing Order**
- **Work Orders**
- **Work Centers**
- **Inventory Movements**
- **Shop Floor**
- **Quality**
- **Scrap**
- **Maintenance**
- **MO Cost vs Real Cost**

المخطط العام:

```text
Product
→ BOM
→ Manufacturing Order
→ Work Orders
→ Work Centers / Shop Floor
→ Material Consumption
→ Production
→ Quality / Scrap
→ Finished Goods
→ MO Cost vs Real Cost
```

| العنصر | دوره في Odoo | لماذا مهم لمستشار ERP؟ |
|---|---|---|
| Product | يمثل المنتج النهائي أو المكون | كل الإنتاج يبدأ من تعريف صنف صحيح |
| BOM | يحدد المكونات والعمليات وربما by-products | يربط المنتج بالمواد والتشغيل والتكلفة |
| Manufacturing Order | أمر التنفيذ الرئيسي | يحول BOM إلى عملية إنتاج فعلية |
| Work Orders | تقسيم MO إلى عمليات تنفيذية | مهم عند وجود مراحل إنتاج متعددة |
| Work Centers | موارد أو أماكن تنفيذ العمليات | تربط التشغيل بالوقت والطاقة والتكلفة |
| Inventory Movements | حركات المكونات والمنتج النهائي | تكشف أثر الإنتاج على المخزون |
| Shop Floor | واجهة تشغيل العامل أو مشرف الإنتاج | تجعل النظام قابلًا للاستخدام على أرض المصنع |
| Quality | فحوصات وتنبيهات جودة | تربط الجودة بالحدث التشغيلي |
| Scrap | تسجيل الهالك أو المرفوض | يمنع اختفاء الفاقد من النظام |
| Maintenance | صيانة المعدات ومراكز العمل | توقف المعدة يؤثر على الجدولة والطاقة |
| MO Cost vs Real Cost | مقارنة المتوقع بالفعلي | أداة عملية لفهم تكلفة الإنتاج دون تعقيد SAP |

## 3. لماذا ندرس Odoo رغم وجود SAP وOracle وMicrosoft؟

دراسة Odoo لا تهدف إلى استبدال SAP أو Oracle كمرجع عميق للتكلفة والإغلاق المالي، بل تهدف إلى فهم نموذج إنتاج عملي قابل للتطبيق بسرعة.

| المجال | ماذا نتعلم من Odoo؟ | ماذا لا نأخذ منه كمرجع عميق؟ |
|---|---|---|
| دورة الإنتاج | كيف نبني دورة سهلة من Product إلى MO إلى Finished Goods | لا نأخذ منه نموذج Enterprise PP كامل |
| BOM | كيف تكون BOM مركزية وسهلة الاستخدام | لا نكتفي به لتعقيدات Cost Object متقدمة |
| Work Orders | كيف نحول العمليات إلى مهام تشغيلية | لا يغطي كل تعقيدات الجدولة الصناعية الثقيلة |
| Shop Floor | كيف نخدم العامل أثناء التنفيذ | لا يغني عن MES عميق في المصانع الكبيرة |
| Quality | كيف نربط الفحص بحدث الإنتاج | لا يغطي كل أنظمة الامتثال المعقدة |
| Costing | MO Cost vs Real Cost عملي وواضح | ليس بديلًا عن WIP/Variance/Settlement العميق |
| Replenishment | Reordering / MTO / MPS كنماذج عملية | ليس تخطيطًا عميقًا مثل SAP أو Microsoft |
| Reporting | تقارير تشغيلية مفيدة | لا يغطي Enterprise Cost Governance بالكامل |

الخلاصة:  
Odoo ممتاز لفهم **كيف نجعل الإنتاج مفهومًا وقابلًا للتشغيل**. أما SAP وOracle فيبقيان أعمق في التكلفة الرسمية، WIP، Variance، Settlement، وPeriod-End Closing.

## 5. BOM في Odoo

في Odoo، **BOM ليست قائمة مواد فقط**. هي العنصر الذي يربط المنتج بمكوناته وعملياته وربما مخرجاته الجانبية.

| مفهوم BOM | دوره | الدرس العملي |
|---|---|---|
| Components | تحدد المواد الداخلة في المنتج | يجب أن تمثل الواقع الفعلي لا التقدير العام |
| Component Quantity | تحدد كمية الاستهلاك القياسية | أي خطأ ينتج استهلاكًا وتكلفة خاطئة |
| Operations | تحدد خطوات التصنيع | BOM يمكن أن تحتوي process وليس مواد فقط |
| Work Centers | ترتبط بالعمليات | التشغيل والتكلفة مرتبطان |
| By-products | مخرجات جانبية متوقعة | لا تخلطها مع Scrap |
| Variants | تخصيص مكونات حسب نسخة المنتج | مهم للمنتجات متعددة الخيارات |
| Manufacturing BOM | تصنيع داخلي | ينتج MO |
| Kit | حزمة أو تجميع تجاري | ليس مثل MO الداخلي |
| Subcontracting BOM | تصنيع لدى طرف خارجي | يحتاج تدفق مواد ومقاول |
| Readiness | جاهزية بدء التصنيع | تؤثر على متى يبدأ التنفيذ |
| MO Cost | BOM تدخل في التكلفة المتوقعة | BOM الخاطئة تعني cost خاطئ |

### ماذا يستفيد مستشار ERP من طريقة Odoo في BOM؟

Odoo يعلم المستشار أن BOM يجب أن تكون **عقدًا تشغيليًا** بين المنتج، المكونات، العمليات، والتكلفة.  
إذا كانت BOM مجرد جدول مواد دون عمليات، وحدات قياس، مخرجات جانبية، أو ارتباط بالتنفيذ، فالنظام سيبقى شكليًا.

## 6. Manufacturing Order في Odoo

**Manufacturing Order — MO** هو أمر التنفيذ الرئيسي في Odoo.

منطقه العملي:

1. اختيار المنتج والكمية.
2. اختيار أو تحميل BOM.
3. ظهور المكونات والعمليات.
4. تأكيد الأمر.
5. تنفيذ التصنيع مباشرة أو عبر Work Orders.
6. تسجيل الإنتاج.
7. تنفيذ Produce All / Mark as Done.
8. دخول المنتج النهائي إلى المخزون.
9. ظهور أثر التكلفة المتوقعة والفعلية.

| مرحلة MO | ماذا يحدث؟ | الأثر على التشغيل والمخزون |
|---|---|---|
| Create | إنشاء أمر تصنيع للمنتج والكمية | لا يكفي دون BOM صحيحة |
| Confirm | تأكيد الأمر واعتماد مكوناته | يبدأ منطق الجاهزية والتنفيذ |
| Components | تحميل المكونات من BOM | يحدد ما سيستهلك |
| Operations | تحميل العمليات من BOM | يحدد Work Orders |
| Start / Execute | بدء التنفيذ أو أوامر العمل | تسجيل الزمن والتقدم |
| Produce All / Mark as Done | إتمام الإنتاج | دخول المنتج النهائي للمخزون |
| Cost Review | مقارنة MO Cost وReal Cost | فهم الفرق بين المتوقع والفعلي |
| Backorder | عند إنتاج جزء من الكمية | يبقي المتبقي في أمر لاحق |

## 7. أنماط التصنيع المخزني في Odoo

Odoo يميز بين ثلاثة أنماط لتدفق التصنيع من ناحية المخزون:

### One-step Manufacturing

يستخدم عندما يكون العميل يريد دورة بسيطة.  
يتم تنفيذ MO دون إظهار تحويلات منفصلة للمكونات والمنتج النهائي بنفس مستوى التفصيل.

مناسب لـ:

- مصانع بسيطة.
- عملاء يريدون تشغيلًا سريعًا.
- حالات لا تحتاج رقابة مخزنية تفصيلية لكل مرحلة.

حدوده:

- رقابة أقل على picking.
- أقل وضوحًا في فصل حركة المكونات عن التخزين النهائي.

### Two-step Manufacturing

يتكون من:

```text
Pick Components
→ Manufacture
```

مناسب عندما يحتاج العميل فصل تجهيز المواد عن تنفيذ الإنتاج.

### Three-step Manufacturing

يتكون من:

```text
Pick Components
→ Manufacture
→ Store Finished Goods
```

مناسب عندما يحتاج العميل رقابة أعلى على خروج المواد ودخول المنتج النهائي.

| النمط | ماذا يحدث؟ | متى يناسب؟ | مستوى الرقابة |
|---|---|---|---|
| One-step | تنفيذ MO مباشرة | مصنع بسيط أو دورة قصيرة | منخفض إلى متوسط |
| Two-step | Pick Components + Manufacture | الحاجة لتجهيز مواد قبل الإنتاج | متوسط |
| Three-step | Pick + Manufacture + Store FG | مستودعات منظمة ورقابة أعلى | أعلى |

### لماذا هذه الفكرة مهمة لنظام ERP محلي مثل ناتج؟

لأنها تمنع خطأ شائعًا: افتراض أن كل العملاء يحتاجون نفس دورة الإنتاج.  
نظام ERP المحلي يجب أن يسمح بدرجات مختلفة من الرقابة، من دورة بسيطة إلى دورة تفصيلية.

## 8. Work Orders وShop Floor

Odoo يحوّل التصنيع من أمر إداري إلى تجربة تشغيلية عبر **Work Orders** وواجهة **Shop Floor**.

| العنصر | ماذا يقدم Odoo؟ | لماذا يفيدنا في فهم التطبيق العملي؟ |
|---|---|---|
| Work Orders | أوامر عمل مرتبطة بعمليات BOM | تفصيل التنفيذ بدل أمر واحد عام |
| Shop Floor | واجهة للعامل أو المشرف | تجعل ERP قريبًا من أرض المصنع |
| Start | بدء العملية وتسجيل الزمن | بداية تتبع فعلي |
| Pause | إيقاف مؤقت | يعكس التوقفات |
| Done | إنهاء أمر العمل | تحديث تقدم العملية |
| Time Tracking | تسجيل زمن التنفيذ | يدخل في Real Cost |
| Work Center Pages | تنظيم العمل حسب مركز | يفيد المشرفين |
| Quality inside operation | فحص داخل العملية | الجودة ليست بعدية فقط |
| Quality Alert | تنبيه من أرض المصنع | مشكلة التشغيل تدخل النظام فورًا |

الدرس المهم:  
النظام الجيد لا يكتفي بإدخال أمر إنتاج، بل يجب أن يخدم العامل ومشرف الإنتاج أثناء التنفيذ.

## 9. Work Centers في Odoo

**Work Centers** تمثل أماكن أو موارد تنفيذ العمليات. وهي ليست مجرد اسم محطة، بل تجمع بين التشغيل والطاقة والزمن والتكلفة.

| إعداد Work Center | أثره على التشغيل | أثره على التكلفة |
|---|---|---|
| Capacity | كم يمكن تشغيله | يؤثر على التخطيط |
| Working Hours | أوقات العمل | يحدد الجدولة |
| Time Efficiency | كفاءة الوقت | يؤثر على توقع مدة العملية |
| OEE Target | مؤشر أداء | يقيس كفاءة المركز |
| Setup Time | وقت التجهيز | يدخل في زمن العملية |
| Cleanup Time | وقت التنظيف | يؤثر على الجدولة |
| Hourly Cost | تكلفة الساعة | تدخل في MO Cost / Real Cost |
| Alternative Work Centers | بدائل عند التعطل | يقلل توقف الإنتاج |
| Allowed Employees | من يعمل على المركز | حوكمة تشغيلية |
| Product-specific Capacity | قدرة حسب المنتج | تخطيط أدق |

الخلاصة:  
Work Center في Odoo هو نقطة ربط بين **المورد، الزمن، القدرة، والتكلفة**.

## 10. التخطيط والتجديد في Odoo

Odoo لا يقدم تخطيطًا عميقًا مثل SAP أو Microsoft، لكنه يقدم أدوات عملية كافية لكثير من الشركات الصغيرة والمتوسطة.

| آلية التخطيط | متى تستخدم؟ | الدرس العملي |
|---|---|---|
| Reordering Rules | للحفاظ على مخزون بين Min/Max | مناسبة للتجديد العملي |
| Make to Order — MTO | عند التصنيع بناءً على طلب محدد | يربط الطلب بالتوريد |
| MPS | للتخطيط على أساس forecast | مناسب لتخطيط متوسط |
| Multilevel BOM | عند وجود subassemblies | يحتاج تخطيطًا لكل مستوى |
| Subassemblies | منتجات نصف مصنعة | يجب تحديد هل تُخزن أم تُنتج عند الحاجة |
| Replenishment | تعويض المخزون | ليس كل شيء يبدأ يدويًا |
| Routes | تحدد Buy/Manufacture/Replenish | ضبطها خطأ يسبب نتائج خاطئة |

الفرق العملي:  
Odoo يقدم تخطيطًا واضحًا وسهلًا، لكنه ليس بديلًا كاملًا عن التخطيط الصناعي العميق الموجود في أنظمة أكبر.

## 11. التتبع Lots / Serial Numbers

| نوع التتبع | متى يستخدم؟ | أثره على التصنيع |
|---|---|---|
| Lot | عند تتبع دفعة من المنتج أو المادة | يربط الدفعة بالإنتاج والجودة |
| Serial Number | عند تتبع كل وحدة منفردة | قد يغير طريقة تنفيذ MO |
| No Tracking | للمواد العادية | أبسط لكن بلا traceability |

أثر التتبع:

- قد يمنع إتمام التصنيع دون إدخال Lot/SN.
- يدعم التحقيقات والجودة والاستدعاء.
- يربط المادة الخام بالمنتج النهائي.
- في حالات serial-controlled multi-unit production قد يحتاج النظام إلى تقسيم أوامر أو منطق خاص لكل وحدة.

إذا كانت تفاصيل تقسيم أوامر serial غير كافية في الملفات:  
**يحتاج تحقق إضافي من مصدر Odoo رسمي أو من تجربة عملية على النظام**.

## 12. الجودة في Odoo

الجودة في Odoo ليست مجرد شكوى بعد الإنتاج، بل يمكن ربطها بالتصنيع وWork Orders وShop Floor.

| عنصر الجودة | دوره | الدرس العملي |
|---|---|---|
| Quality Control Point | قاعدة توليد فحص | الجودة يجب أن تكون event-based |
| Quality Check | فحص فعلي | يظهر أثناء العملية أو عند حدث محدد |
| Quality Alert | تسجيل مشكلة جودة | يجب ألا تبقى المشاكل خارج النظام |
| Work Order Operation | ربط الفحص بعملية | فحص أثناء التشغيل |
| Shop Floor Quality | تنفيذ الفحص من أرض المصنع | يقلل الفجوة بين العامل والنظام |
| Pass/Fail | نتيجة فحص بسيطة | مناسبة للعمليات السريعة |
| Measure | قياس | مناسب لفحوص كمية |
| Take Picture | صورة كدليل | مفيد للتوثيق |
| Usage / Decision Logic | قبول أو رفض | يحتاج تحقق إضافي من مصدر Odoo رسمي أو من تجربة عملية على النظام |

الدروس العملية:

- الجودة يجب أن ترتبط بحدث: تصنيع، عملية، استلام، أو تشغيل.
- الجودة ليست فقط بعد الإنتاج.
- Quality Alert مهم لتوثيق المشكلة ومتابعتها.

## 13. Scrap وBy-products

| البند | Scrap | By-product |
|---|---|---|
| المعنى | هالك أو مرفوض أو فاقد | مخرج جانبي متوقع من التصنيع |
| متى يحدث؟ | أثناء التصنيع أو عند فشل مادة/منتج | عند إكمال MO حسب BOM |
| هل هو مقصود؟ | لا، غالبًا غير مقصود | نعم، معروف مسبقًا |
| أين يعرف؟ | كحركة Scrap أو من Shop Floor/Manufacturing | على BOM |
| أثره على المخزون | ينقل إلى Scrap Location | يدخل كمخزون لمخرج جانبي |
| أثره على التكلفة | يعكس فاقدًا أو رفضًا | قد يؤثر على فهم تكلفة العملية |
| الدرس | لا تخفِ الهالك | لا تعتبر كل مخرج جانبي هالكًا |

### لماذا الخلط بين Scrap وBy-product خطأ تحليلي؟

لأن Scrap يعني فاقدًا أو مرفوضًا يجب مراقبته، بينما By-product مخرج متوقع من العملية.  
خلطهما يؤدي إلى قراءة خاطئة للمخزون والتكلفة وجودة العملية.

---

## 14. Maintenance وربطها بالإنتاج

Odoo يربط الصيانة بالإنتاج من خلال Work Centers والمعدات والتوقفات.

| عنصر الصيانة | أثره على الإنتاج |
|---|---|
| Maintenance Request | تسجيل طلب صيانة قد يؤثر على المورد |
| Equipment | أصل أو آلة مرتبطة بالتشغيل |
| Work Center Time Off | وقت توقف مركز العمل |
| Blocking Work Center | منع جدولة العمل على مركز متوقف |
| Alternative Work Center | تحويل العمل لمورد بديل |
| Maintenance from Shop Floor | ربط المشكلة بأرض المصنع |
| Downtime | يؤثر على الجدولة والطاقة |

الدرس العملي:  
الصيانة ليست نظامًا منفصلًا عن الإنتاج. إذا تعطلت آلة أو Work Center، يجب أن يتأثر التخطيط والتنفيذ.

## 15. Subcontracting في Odoo

Odoo يدعم تصنيعًا خارجيًا من خلال **Subcontracting BOM** ومسارات تزويد المقاول.

| المفهوم | ماذا يعني؟ | الدرس العملي |
|---|---|---|
| Subcontracting BOM | BOM خاصة بتصنيع خارجي | ليس كل تصنيع يتم داخليًا |
| Subcontractor | طرف خارجي يصنع المنتج | يجب ربطه بالتوريد |
| Resupply Subcontractor | إرسال مكونات للمقاول | يجب تتبع المواد الخارجة |
| Dropship Subcontractor | إرسال مباشر حسب السيناريو | يحتاج ضبط مسار |
| Finished Product Receipt | استلام المنتج النهائي | يربط التصنيع الخارجي بالمخزون |
| Costing Impact | تكلفة المقاول والمكونات | يحتاج تحقق إضافي من مصدر Odoo رسمي أو من تجربة عملية على النظام |

Subcontracting مهم لأي عميل لديه تصنيع خارجي، لأن النظام يجب أن يفرق بين شراء منتج جاهز وتصنيع منتج لدى مقاول باستخدام مكونات.

## 16. تكلفة الإنتاج في Odoo

هذا القسم مهم، لكن لا يجب تحويله إلى SAP.

### MO Cost

تكلفة تقديرية مبنية على:

- BOM Components.
- Operations.
- Work Centers.
- Hourly Cost.
- Estimated duration.

### Real Cost

تكلفة فعلية مبنية على:

- الاستهلاك الفعلي.
- الوقت الفعلي.
- تكلفة مراكز العمل.
- تكلفة الموظفين أو العاملين عند توفرها.

| المفهوم | ماذا يعني في Odoo؟ | ما الدرس العملي؟ |
|---|---|---|
| MO Cost | تكلفة متوقعة لأمر التصنيع | تعطي معيارًا للمقارنة |
| Real Cost | التكلفة الفعلية بعد التنفيذ | تكشف الفرق عن المتوقع |
| Component Cost | تكلفة المكونات | BOM تؤثر مباشرة |
| Work Center Cost | تكلفة وقت مركز العمل | الزمن والتكلفة مرتبطان |
| Employee Cost | تكلفة عامل إن توفرت | يزيد دقة Real Cost |
| Standard Cost | طريقة تقييم ثابتة | ليست نفس MO Cost |
| AVCO | متوسط تكلفة | تقييم مخزون |
| FIFO | طبقات واردة | تقييم مخزون |
| Inventory Valuation | تقييم محاسبي للمخزون | منفصل عن MO Cost التشغيلي |
| WIP / Settlement | ليس بنفس عمق SAP | لا تبالغ في تحميل Odoo ما ليس هدفه |

### ما الذي يجب ألا نبالغ فيه عند دراسة تكلفة Odoo؟

لا يجب اعتبار Odoo مرجعًا عميقًا في:

- WIP الرسمي المتقدم.
- Variance المحاسبي التفصيلي.
- Settlement على طريقة SAP.
- Cost Object Controlling.
- Period-End Closing المعقد.

قوة Odoo هنا هي **وضوح MO Cost vs Real Cost**، وليس عمق التكلفة المؤسسية.

## 18. أهم دروس Odoo لمستشار ERP

| الدرس | كيف أستخدمه في عملي؟ |
|---|---|
| لا تبدأ الإنتاج دون Product وBOM صحيحين | اجعل Master Data شرطًا قبل UAT |
| BOM يجب أن تمثل الواقع | لا تقبل BOM شكلية |
| Work Orders ضرورية عند تعدد المراحل | لا تجعل MO العام يغطي كل شيء |
| Shop Floor مهم للاستخدام | فكر في العامل وليس المدير فقط |
| Work Center يجمع التشغيل والزمن والتكلفة | اربط الموارد بالتكلفة |
| MO Cost vs Real Cost فكرة ممتازة | استخدمها لتعليم العملاء الفرق بين المتوقع والفعلي |
| Quality يجب أن ترتبط بحدث | لا تجعل الجودة شكوى منفصلة فقط |
| Scrap وBy-product مختلفان | افصل الفاقد عن المخرجات الجانبية |
| Maintenance تؤثر على الطاقة | لا تفصل الصيانة عن الجدولة |
| Subcontracting يحتاج تدفق مواد واضح | لا تعامله كمشتريات عادية فقط |
| One/Two/Three-step مهم | صمم مستوى الرقابة حسب العميل |
| Traceability يغير طريقة التنفيذ | لا تضفه كحقل شكلي |

## 19. ما الذي لا يكفي فيه Odoo للدراسة العميقة؟

| المجال | لماذا لا نعتمد على Odoo وحده؟ | المرجع الأفضل |
|---|---|---|
| WIP العميق | Odoo يعرض منطقًا عمليًا أخف | SAP |
| Variance الرسمي | لا يقدم نفس عمق تصنيفات SAP/Oracle | SAP / Oracle |
| Settlement | ليس محورًا عميقًا مثل SAP | SAP |
| Cost Object Controlling | غير بنفس العمق | SAP |
| Cost Distributions | أعمق في Oracle | Oracle |
| Work Order Close المحاسبي | Oracle أوضح في cost events | Oracle |
| Period-End Closing المتقدم | ليس محور Odoo الأساسي | SAP / Oracle |
| Enterprise Manufacturing Governance | يحتاج أنظمة أعمق | SAP / Microsoft |
| Advanced Planning | Odoo أبسط | Microsoft / SAP |
| Mixed-mode Manufacturing كبير | يحتاج حوكمة أوسع | Microsoft / SAP / Oracle |

لا يعني ذلك أن Odoo ضعيف، بل يعني أن استخدامه كمرجع يجب أن يكون صحيحًا:  
**Odoo للتشغيل العملي وتجربة المستخدم، SAP/Oracle للتكلفة والإغلاق المالي العميق.**

## 20. ملخص تنفيذي

Odoo مفيد جدًا كنموذج إنتاج عملي ومبسط. قوته ليست في تعقيد التكلفة مثل SAP، بل في تحويل مفاهيم الإنتاج إلى تجربة تشغيل مفهومة:

- Product واضح.
- BOM عملية.
- Manufacturing Order مباشر.
- Work Orders عند وجود مراحل.
- Work Centers تربط الزمن والتكلفة.
- Shop Floor يخدم العامل والمشرف.
- Quality مرتبطة بالتشغيل.
- Scrap وBy-products مفهومان منفصلان.
- Maintenance تؤثر على الطاقة.
- Subcontracting مدعوم كتصنيع خارجي.
- MO Cost vs Real Cost يعطي رؤية عملية للتكلفة.

بالنسبة لنظام ERP محلي مثل **ناتج**، أهم درس من Odoo هو:

> لا تبنِ الإنتاج كنماذج إدخال فقط. ابنِه كتدفق عملي يبدأ من المنتج وBOM، يمر بأمر التصنيع وWork Orders وShop Floor، وينتهي بمنتج نهائي وتكلفة فعلية وتقارير رقابية.

Odoo لا يكفي وحده لفهم WIP وVariance وSettlement العميق، لكنه ممتاز لفهم كيف يجب أن يبدو نظام إنتاج عملي قابل للاستخدام.
