# الدرس 05: علامات الفشل المبكر بعد Go Live

## لماذا هذا الدرس مهم

لأن كثيرًا من المشاكل بعد Go-Live لا تبدأ ككارثة كبيرة، بل كإشارات صغيرة تتكرر يوميًا. المستشار الضعيف يراها ضوضاء طبيعية. المستشار الجيد يراها Early Warning Signals تكشف أن النظام لم يستقر بعد أو أن التصميم لا يعكس الواقع بالكامل.

## الهدف من الدرس

بعد هذا الدرس يجب أن تكون قادرًا على:

- رصد علامات الخطر المبكرة بعد التشغيل.
- التمييز بين مشكلة تدريب ومشكلة بيانات ومشكلة تصميم.
- فهم ما الذي يجب مراقبته يوميًا في الأسابيع الأولى.
- بناء عقلية دعم واستقرار لا مجرد تسليم مشروع.

## الشرح المفاهيمي الأساسي

بعد Go-Live، لا تفترض أن صمت المستخدمين يعني نجاحًا. راقب إشارات مثل:

- كثرة الأوامر العالقة
- أخطاء GI أو GR أو Backflush
- WIP غير طبيعي
- Variances غير مفهومة
- شكاوى متكررة أن النظام لا يعكس الواقع
- تقارير لا يعتمد عليها المستخدمون

### أمثلة على إشارات مبكرة مهمة

| الإشارة | ماذا قد تعني؟ |
| --- | --- |
| أوامر كثيرة مفتوحة بلا إغلاق | ضعف حوكمة أو training أو process gap |
| تكرار أخطاء الصرف | BOM أو stock setup أو backflush issue |
| Lots أو Serials مفقودة | Traceability design أو discipline ضعيف |
| WIP يرتفع دون تفسير | تأخر close أو completion أو cost processing |
| المستخدمون يعملون خارج النظام | process not trusted أو gap حقيقي |

### لماذا هذه الإشارات مهمة؟

لأن معالجتها مبكرًا أقل كلفة كثيرًا من تركها حتى تتحول إلى:

- فقدان ثقة كامل بالنظام
- تراكم بيانات خاطئة
- تقارير غير موثوقة
- صعوبة شديدة في stabilization

## كيف يظهر المفهوم في الأنظمة الثلاثة

### Dynamics 365

في Dynamics تظهر الإشارات المبكرة غالبًا عبر أوامر غير منتهية، حركات صرف أو استلام غير منضبطة، Quality issues، وVariances أو تقارير لا تطابق الواقع. كل ذلك يجب قراءته كإشارة إلى data أو process gap لا كأخطاء منفصلة فقط.

### Oracle Fusion Cloud

في Oracle تظهر الإشارات أيضًا عبر Pending transactions وCost Distributions غير مكتملة وWork Orders غير مغلقة وQuality exceptions وreports ضعيفة. وهذا يجعل مراقبة أول شهر تشغيل مسألة أساسية لا اختيارية.

### SAP S/4HANA

في SAP تصبح بعض الإشارات أوضح جدًا، مثل أخطاء الحركات التلقائية أو مشاكل backflush أو أوامر لا تسوى أو تقارير MRP أو WIP أو variance غير مفهومة. وهذه إشارات يجب التعامل معها بسرعة قبل أن تتوسع.

## منظور المستشار Functional Consultant View

في أول أسابيع التشغيل، راقب يوميًا:

- الأوامر المفتوحة والقديمة
- أوامر completed not closed
- استثناءات الصرف والاستلام
- تقارير الجودة والاستثناءات
- WIP وVariances العالية
- أكثر الشكاوى تكرارًا من المستخدمين

ثم حاول دائمًا تصنيف السبب إلى:

- Training
- Data
- Configuration
- Real gap

## الأخطاء الشائعة Common Mistakes

- اعتبار مشاكل أول أسبوعين أمرًا عاديًا بالكامل.
- معالجة كل شكوى كحادثة منفصلة دون رؤية نمط متكرر.
- تجاهل إشراك المالية أو المستودعات في stabilization.
- الاكتفاء بالدعم التفاعلي دون تقارير مراقبة استباقية.
- الدفاع عن التصميم بدل التحقيق في الواقع.

## المراجعة السريعة Quick Review

- المشاكل المبكرة غالبًا تظهر كأنماط لا كحوادث منفردة فقط.
- سرعة الرصد والتصنيف أهم من سرعة التبرير.
- أول شهر تشغيل هو فترة تثبيت حاسمة.
- المستشار الجيد يبني رقابة يومية على الإشارات المبكرة.

## مهمة الدرس Study Task

اكتب قائمة من ثماني إشارات خطر تراها مهمة جدًا بعد Go-Live لموديول الإنتاج، ثم صنف كل واحدة إلى:

- Data issue
- Process issue
- Training issue
- Gap issue
